01 — Problema

El proceso ya existe.
El formato es el problema

Los equipos de Buk ya documentan decisiones y tienen ownership definido. El problema no es la cultura — es que Google Docs no es un formato que la IA pueda consumir de manera confiable, estructurada y revisada.

Hoy — Google Docs
  • ✕El contexto existe pero es caro de consumir. Cada desarrollador pasa a la IA lo que estima necesario, sin que ese recorte haya sido revisado.
  • ✕La información no está unificada. Un doc puede tener la decisión correcta pero el agente nunca la verá si el dev no la incluyó en su prompt.
  • ✕Los docs crecen. Con el tiempo, leer el contexto completo de un feature se vuelve una carga real — tanto para personas como para la IA.
  • ✕El ownership y las decisiones existen en el doc, pero no hay un contrato claro de qué lee quién antes de implementar.
  • ✕Trabajar con IA desde un doc largo es difícil de estandarizar. Cada dev tiene su propia forma de resumir y pasar contexto.
Con SDD — Markdown en Git
  • ✓El contexto está dividido en archivos con propósito único. El agente sabe exactamente qué leer y en qué orden — sin depender de cada dev.
  • ✓El recorte de contexto es parte del diseño, no del criterio individual. Las dependencias entre misiones determinan qué se carga y qué no.
  • ✓Los archivos son pequeños por convención. Un 1_mission.md bien escrito es denso pero acotado — fácil de leer para personas y para la IA.
  • ✓Ownership y reviewers definidos en cada archivo. La revisión ocurre antes de que el agente ejecute, no después.
  • ✓El flujo de contexto es reproducible. Cualquier dev — o agente — que siga el orden de lectura llega al mismo punto de partida.

El problema no es la falta de documentación — es que el formato actual no fue diseñado para ser consumido por agentes de IA de forma estructurada, predecible y revisada.

El contexto no solo es caro para la IA, sino también para cada desarrollador

03 — Roles

Cada archivo tiene un dueño

La responsabilidad de cada documento está asignada. Saber quién escribe y quién aprueba elimina ambigüedad y acelera el proceso de revisión.

04 — Agentes

IA que lee las specs,
no que las inventa

Los agentes no improvisan. Leen el contexto en orden definido, preguntan antes de actuar, y producen artefactos que quedan versionados en el repo.

El orden de lectura importa

Antes de escribir una línea de código, el agente lee el contexto en este orden exacto. El contexto acumulado es lo que permite implementar sin ambigüedad.

06 — Architecture Decision Records

Las decisiones
no se olvidan

Cada decisión técnica significativa queda documentada en el nivel más bajo que aplique. Un ADR aceptado debe reflejarse en el track.md o 1_mission.md correspondiente.

Nivel global

Todos los equipos

decisions/YYYYMMDD_slug.md

Convenciones de naming, workspace de agentes, patrones transversales a toda la org.

Nivel equipo

Un equipo

teams/<team>/decisions/

Decisiones que afectan solo a ese equipo: patrones de testing, convenciones de código.

Nivel track

Todas las misiones del track

teams/<team>/tracks/YYYY/<track>/ADR/

Decisiones que afectan cómo se implementan todas las misiones de ese track.

Nivel misión

Una misión específica

...<track>/<mission>/ADR/

Incluye siempre un ADR de alternativas de solución (alto nivel) como punto de partida.

# Formato de un ADR — _templates/adr.md Status: proposed | accepted | deprecated | superseded Date: YYYY-MM-DD Track: 0513_nombre-del-track Mission: 01_nombre-mision # opcional si es a nivel track ## Contexto ¿Qué situación o restricción motiva esta decisión? ## Decisión ¿Qué se decidió hacer? ## Alternativas consideradas - Opción A: [descripción] — descartada porque... - Opción B: [descripción] — descartada porque... ## Consecuencias Qué implica esta decisión en términos de mantenimiento, performance, deuda técnica.
07 — Estructura del repositorio

Un archivo para cada cosa

La estructura de carpetas es la navegación. Cada archivo tiene un propósito único, un dueño, y un reviewer definido.

sdd-buk-docs/ ├── _templates/ # Plantillas globales — no se sobreescriben por equipo ├── agents/ # Prompts de los agentes de IA │ ├── orquestador.md # Punto de entrada — Haiku │ ├── investigacion.md │ ├── armado.md │ ├── ejecutor.md │ ├── arquitecto.md │ └── cloud-architect.md ├── decisions/ # ADRs globales └── teams/ └── <equipo>/ ├── context.md # Board Jira + pack buk-webapp ├── decisions/ └── tracks/ └── YYYY/MMDD_<track>/ ├── spec-track.md # Owner: PM ├── track.md # Owner: Arquitecto ├── ADR/ └── NN_<misión>/ ├── spec-mission.md # Owner: PM ├── 1_mission.md # Owner: Champion ├── 2_jira-cards.md ├── 3_summary.md └── ADR/ └── 00_alternativas_de_solucion.md

08 — Comandos rápidos

Empezar en segundos

Todos los agentes corren desde buk-webapp con las specs accesibles via symlink en specs/. Las skills detectan paths, leen contexto y confirman antes de escribir.

/sdd-track <equipo> "<problema>"
Crea track nuevo. Explora el codebase, genera spec-track.md + track.md borradores y crea la estructura de carpetas.
investigacion
/sdd-mission <equipo> <track> "<objetivo>"
Crea misión dentro de un track existente. Detecta número de secuencia y genera spec-mission.md + 1_mission.md + 2_jira-cards.md.
armado
/sdd-execute <equipo> <track>
Implementa la misión en estado ready. Si hay varias, pregunta cuál. Lee el stack completo de specs antes de escribir código.
ejecutor
/sdd-adr <equipo> [<track>] "<decisión>"
Documenta una decisión técnica. Lee ADRs existentes en ese scope y sus padres, explora el codebase, genera borrador con alternativas.
arquitecto

¿No sé en qué estado está el track? El Orquestador lee todos los archivos, detecta el próximo paso, y pregunta: ¿Ejecuto? (s/n)
